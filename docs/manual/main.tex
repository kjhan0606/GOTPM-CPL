%======================================================
\chapter{Introduction}
\label{ch:intro}
%======================================================

GOTPM-CPL (Grid-of-Oct-Tree Particle Mesh), also referred to as PSC3 (Parallel Simulation Code~3), is a massively parallel cosmological N-body simulation code that computes the gravitational evolution of dark matter particles in an expanding universe.  The code implements the hybrid TreePM algorithm: long-range forces are computed on a uniform mesh using FFT-based Poisson solvers (PM), while short-range corrections are applied through a Barnes--Hut-type tree walk.

\section{Key Features}

\begin{itemize}[leftmargin=2em]
  \item \textbf{Hybrid TreePM force calculation} --- PM for long-range + Tree for short-range correction
  \item \textbf{CPL dark energy models} --- Full support for the Chevallier--Polarski--Linder parameterization $w(a) = w_0 + w_a(1-a)$, including Poisson correction factors
  \item \textbf{2LPT initial conditions} --- Built-in 2nd-order Lagrangian perturbation theory for accurate IC generation
  \item \textbf{MPI + OpenMP parallelization} --- Slab domain decomposition with hybrid parallelism
  \item \textbf{Variable PM grid (VarPM)} --- Adaptive z-domain boundaries for load balancing
  \item \textbf{Index-based position encoding} --- High-precision particle positions via displacement + Lagrangian index
  \item \textbf{Custom memory allocator} --- Stack-based pool allocator for deterministic memory management
  \item \textbf{Friends-of-Friends (FoF)} --- Built-in parallel group finder
  \item \textbf{Power spectrum measurement} --- On-the-fly $P(k)$ computation with TSC deconvolution
\end{itemize}

\section{Algorithm Overview}

The simulation proceeds through the following steps at each time step:

\begin{enumerate}[leftmargin=2em]
  \item \textbf{PM force} --- Assign particle mass to grid via TSC, solve Poisson equation in Fourier space, interpolate forces back to particles (4th-order FDA).
  \item \textbf{Tree correction} --- Build cell-linked-list, compute short-range force corrections using spline-softened gravity with opening angle $\theta$.
  \item \textbf{Velocity kick} --- Update velocities using leap-frog integration.
  \item \textbf{Position drift} --- Advance particle positions.
  \item \textbf{Migration} --- Exchange particles between MPI domains if they have crossed slab boundaries.
  \item \textbf{Domain rebalancing} --- Adjust slab boundaries if load imbalance exceeds threshold.
\end{enumerate}

\begin{important}
The code uses a \emph{split-step} (leap-frog) integrator where velocities and positions are offset by half a time step.  The \code{halfstep} structure controls the synchronization of positions and velocities for output.
\end{important}

\chapter{Quick Start Guide}
\label{ch:quickstart}
%======================================================

This chapter provides a step-by-step guide to building and running the code.

\section{Prerequisites}

\begin{itemize}[leftmargin=2em]
  \item \textbf{Intel C/Fortran Compilers} --- \code{mpiicx} and \code{mpiifx} (or \code{mpiicc}/\code{mpiifort})
  \item \textbf{FFTW3} --- Single-precision with MPI and OpenMP thread support (\code{libfftw3f}, \code{libfftw3f\_mpi}, \code{libfftw3f\_omp})
  \item \textbf{MPI} --- Intel MPI or OpenMPI
  \item \textbf{OpenMP} --- Enabled via \code{-qopenmp}
\end{itemize}

\section{Building the Code}

\begin{codebox}[Build Commands]
\begin{lstlisting}[style=bash,numbers=none]
cd PSC3_CPL_mv_new
# Edit Rules.make for your system (compiler, FFTW path, NMEG)
make clean
make this
# Produces: namu.exe
\end{lstlisting}
\end{codebox}

The build system consists of two files:
\begin{itemize}[leftmargin=2em]
  \item \file{Rules.make} --- System-specific compiler settings, library paths, and compile flags.
  \item \file{Makefile} --- Object file list and build targets.
\end{itemize}

\begin{remark}[Compiler Selection]
The primary target compiler is Intel \code{mpiicx}/\code{mpiifx}.  For PGI or GCC compilers, use the corresponding \file{Rules.make.*} variant (e.g., \file{Rules.make.pgcc}).
\end{remark}

\section{Setting Up a Run}

\subsection{Directory Structure}

Create a separate run directory:

\begin{codebox}[Run Directory Setup]
\begin{lstlisting}[style=bash,numbers=none]
mkdir MV_Om0.31_-1.2_-2.4
cd MV_Om0.31_-1.2_-2.4
ln -s ../PSC3_CPL_mv_new/namu.exe .
cp ../PSC3_CPL_mv_new/params.dat .
# Edit params.dat for your cosmological model
\end{lstlisting}
\end{codebox}

\subsection{Parameter File}

Edit \file{params.dat} to specify cosmological parameters, grid resolution, and run settings.  See Chapter~\ref{ch:params} for the full parameter reference.

\subsection{Input Power Spectrum}

Prepare an ASCII power spectrum file (e.g., from CAMB) with two columns: $k$ [$h$/Mpc] and $P(k)$ [$({\rm Mpc}/h)^3$].  Set \code{Powerflag = 2} and provide the filename in \code{Ascii Powerfile}.

\subsection{Job Submission (SLURM)}

\begin{codebox}[Example SLURM Job Script]
\begin{lstlisting}[style=bash,numbers=none]
#!/bin/bash
#SBATCH --job-name=MV311224
#SBATCH --ntasks=16
#SBATCH --cpus-per-task=4
#SBATCH --time=1000000:00
#SBATCH --output outfile.o%j.grammar
#SBATCH --error errfile.e%j.grammar
#SBATCH -p normal

export OMP_NUM_THREADS=$SLURM_CPUS_PER_TASK
srun ./namu.exe params.dat
\end{lstlisting}
\end{codebox}

\begin{infobox}[Choosing MPI Ranks and Threads]
The number of MPI ranks (\code{ntasks}) determines the z-slab decomposition.  The FFTW3 domain decomposition requires that $N_z$ is divisible by the number of MPI ranks.  Each rank spawns \code{cpus-per-task} OpenMP threads for shared-memory parallelism within the tree correction and TSC assignment.
\end{infobox}

\section{Particle Types}

The code uses two particle structures that share the same memory through pointer casting.  This is the core of the hybrid PM+Tree design.

\subsection{PM Particle (\code{pmparticletype})}

\begin{definition}[pmparticletype --- 32 bytes]
\begin{lstlisting}[style=ccode,numbers=none]
typedef struct pmparticletype {
    float x, y, z;         /* displacement from index position */
    float vx, vy, vz;      /* peculiar velocity */
    long long indx;        /* Lagrangian grid index */
} pmparticletype;
\end{lstlisting}
The PM particle stores only displacement and velocity.  The true position is reconstructed via:
\[
  X = \text{fmod}\!\Big(x_\text{disp} + (\text{indx} \bmod m_x) + N_x,\; N_x\Big)
\]
where $m_x = N_x / N_\text{space}$.  This encoding preserves sub-grid precision even for $N \geq 1024^3$.
\end{definition}

\subsection{Tree Particle (\code{treeparticletype})}

\begin{definition}[treeparticletype --- 40 bytes]
\begin{lstlisting}[style=ccode,numbers=none]
typedef struct treeparticletype {
    float x, y, z;
    float vx, vy, vz;
    long long indx;
    struct treeparticletype *next;  /* linked-list pointer */
} treeparticletype;
\end{lstlisting}
Extends \code{pmparticletype} with a \code{next} pointer for the cell-linked-list tree structure.  The conversion between the two types is done in-place via backward copy (\code{pm2treetype}) and forward copy (\code{tree2pmtype}).
\end{definition}

\begin{important}
The in-place conversion between \code{pmparticletype} and \code{treeparticletype} requires careful handling because both pointers alias the same memory.  The code uses \code{memmove} through \code{char*} pointers to avoid strict-aliasing violations that can cause data corruption at \code{-O2} optimization.
\end{important}

\subsection{Position Reconstruction Macros}

\begin{lstlisting}[style=ccode,numbers=none]
/* With -DXYZDBL -DINDEX: */
#define XofP(p) fmod((double)((p)->x) + (double)(((p)->indx % simpar.mx)) + simpar.nx, simpar.nx)
#define YofP(p) fmod((double)((p)->y) + (double)((((p)->indx % simpar.mxmy) / simpar.mx)) + simpar.ny, simpar.ny)
#define ZofP(p) fmod((double)((p)->z) + (double)(((p)->indx / simpar.mxmy)) + simpar.nz, simpar.nz)
\end{lstlisting}

\section{Simulation Parameters (\code{SimParameters})}

\begin{definition}[SimParameters struct]
The global \code{simpar} structure holds all runtime parameters:
\begin{itemize}[leftmargin=2em]
  \item \textbf{MPI identity}: \code{myid}, \code{nid}
  \item \textbf{Cosmology}: \code{omep} ($\Omega_m$), \code{omeplam} ($\Omega_\Lambda$), \code{wlam0} ($w_0$), \code{wlam1} ($w_a$), \code{CsDE} (sound speed)
  \item \textbf{Grid}: \code{nx}, \code{ny}, \code{nz}, \code{mx}, \code{my}, \code{mz}, \code{mxmy}
  \item \textbf{Scale factors}: \code{amax}, \code{anow}, \code{astep}
  \item \textbf{Force}: \code{theta} (opening angle), \code{pcorr} (Poisson correction)
  \item \textbf{Domain}: \code{local\_nz}, \code{local\_z\_start}, \code{zmin}, \code{zmax}
\end{itemize}
\end{definition}

\section{Memory Management}

\begin{definition}[Stack-Based Memory Pool]
The code allocates a single large memory block at startup (\code{NMEG} megabytes) and manages sub-allocations through a stack-based allocator defined in \file{Memory2.c}:
\begin{itemize}[leftmargin=2em]
  \item \code{Malloc(size, PPTR(var))} --- Push allocation onto stack
  \item \code{Free(ptr)} --- Pop or compact stack
  \item \code{Realloc(ptr, size)} --- Resize with automatic compaction of subsequent allocations
  \item \code{freespace()} --- Query available memory
\end{itemize}
The allocator tracks each allocation's size, starting address, and a pointer-to-pointer back-reference, enabling automatic pointer updates when memory is compacted.
\end{definition}

\begin{remark}[Realloc Macro]
The header \file{Memory.h} defines:
\begin{lstlisting}[style=ccode,numbers=none]
#define Realloc(A,B) Realloc(A,B,PPTR(A))
\end{lstlisting}
This macro automatically supplies the third argument (pointer-to-pointer), so user code simply calls \code{Realloc(ptr, newsize)}.
\end{remark}

\label{ch:ic}
%======================================================

\section{Zel'dovich Approximation (1st-Order LPT)}

The Zel'dovich approximation displaces particles from a uniform grid according to the linear displacement field:
\begin{equation}
  \mathbf{x}(\mathbf{q}, a) = \mathbf{q} - \frac{D_1(a)}{a}\,\nabla\phi^{(1)}(\mathbf{q})
\end{equation}
where $\mathbf{q}$ is the Lagrangian coordinate (stored as \code{indx}), $D_1(a)$ is the linear growth factor, and $\phi^{(1)}$ is the gravitational potential from the linear density field.

The velocity is:
\begin{equation}
  \mathbf{v} = \dot{a}\,\mathbf{x} - a\,\dot{D}_1\,\nabla\phi^{(1)}
\end{equation}

\section{2nd-Order LPT}

The 2LPT adds a second-order correction:
\begin{equation}
  \mathbf{x}(\mathbf{q}, a) = \mathbf{q} - D_1(a)\,\boldsymbol{\Psi}^{(1)}(\mathbf{q}) + D_2(a)\,\boldsymbol{\Psi}^{(2)}(\mathbf{q})
\end{equation}
where $D_2 \approx -\frac{3}{7}D_1^2$ for $\Omega_m = 1$, and
\begin{equation}
  \nabla^2\phi^{(2)} = \sum_{i>j}\left(\phi^{(1)}_{,ii}\,\phi^{(1)}_{,jj} - (\phi^{(1)}_{,ij})^2\right)
\end{equation}

The 2LPT initial conditions are significantly more accurate than Zel'dovich for moderate-to-high redshift starts ($z_i \sim 50$).

\begin{algorithm}[IC Generation]
\begin{enumerate}[leftmargin=2em]
  \item Generate Gaussian random field $\delta(\mathbf{k})$ using input $P(k)$ and random seed.
  \item Compute 1st-order potential: $\tilde\phi^{(1)}(\mathbf{k}) = -\tilde\delta(\mathbf{k})/k^2$.
  \item (2LPT only) Compute 2nd-order source and potential $\phi^{(2)}$.
  \item Assign displacements and velocities to particles on a uniform grid.
  \item Migrate particles to their correct MPI domains.
\end{enumerate}
\end{algorithm}


%======================================================
\chapter{PM Force Calculation}
\label{ch:pm}
%======================================================

\section{TSC Mass Assignment}

The Triangular Shaped Cloud scheme assigns particle mass to the $3^3 = 27$ nearest grid points using a quadratic spline window:
\begin{equation}
  W_\text{TSC}(x) = \begin{cases}
    \frac{3}{4} - x^2 & |x| \leq \frac{1}{2} \\[4pt]
    \frac{1}{2}\left(\frac{3}{2} - |x|\right)^2 & \frac{1}{2} < |x| \leq \frac{3}{2} \\[4pt]
    0 & |x| > \frac{3}{2}
  \end{cases}
\end{equation}

The density field is:
\begin{equation}
  \rho(\mathbf{x}_g) = \frac{m_p}{\Delta x^3}\sum_p W_\text{TSC}(\mathbf{x}_g - \mathbf{x}_p)
\end{equation}

The implementation in \file{tsc\_omp2.c} uses OpenMP parallelization with thread-local density arrays to avoid race conditions.

\section{FFT-Based Poisson Solver}

\begin{algorithm}[PM Force Pipeline]
\begin{enumerate}[leftmargin=2em]
  \item \textbf{Density field} $\rho(\mathbf{x})$ computed via TSC.
  \item \textbf{Forward FFT}: $\tilde\rho(\mathbf{k}) = \mathcal{F}[\rho]$ using FFTW3-MPI (\file{fft.c}).
  \item \textbf{Poisson solve}: $\tilde\phi(\mathbf{k}) = -\tilde\rho(\mathbf{k})/k^2$ (\file{p\_solver.F}).
  \item \textbf{Dark energy correction}: $\tilde\phi \to p_\text{corr}(a)\cdot\tilde\phi$ for CPL models.
  \item \textbf{Backward FFT}: $\phi(\mathbf{x}) = \mathcal{F}^{-1}[\tilde\phi]$.
  \item \textbf{Force interpolation}: 4th-order FDA gives $\mathbf{F} = -\nabla\phi$ at particle positions (\file{fda4.c}).
\end{enumerate}
\end{algorithm}

\section{4th-Order FDA}

The force at grid point $i$ is:
\begin{equation}
  F_i = \frac{1}{12\Delta x}\left(-\phi_{i+2} + 8\phi_{i+1} - 8\phi_{i-1} + \phi_{i-2}\right)
\end{equation}
This requires padding the potential mesh by 3 slices below and 4 slices above the local domain, handled by \code{paddingpotentialmesh()} in \file{VarPM.c}.

\section{Variable PM (VarPM)}

With \flag{VarPM} enabled, each MPI rank can own a non-uniform number of z-slabs.  The z-domain boundaries (\code{zmin}, \code{zmax}) adapt to balance particle counts.  The TSC and FDA routines accept variable z-boundaries and convert between the local variable-width domain and the global uniform FFTW layout via \code{tsc\_fftw\_mesh()} in \file{VarPM.c}.


%======================================================
\chapter{Tree Force Correction}
\label{ch:tree}
%======================================================

\section{Cell-Linked-List Construction}

The tree correction uses a cell-linked-list rather than a full octree.  In \code{ParticleCellLinkedList()} (\file{kjhtpm.mod1.gpu.omp.c}):

\begin{enumerate}[leftmargin=2em]
  \item Divide the local domain into cells of width $\sim 4$ grid spacings.
  \item Assign each particle to its cell using integer truncation of position.
  \item Build a linked list through the \code{next} pointer of \code{treeparticletype}.
\end{enumerate}

\begin{remark}[Index Clamping]
Particle positions near domain boundaries can produce out-of-range cell indices.  The code clamps indices to valid range:
\begin{lstlisting}[style=ccode,numbers=none]
if(ix < 0) ix = 0;  if(ix >= mxm1) ix = mxm1;
if(iy < 0) iy = 0;  if(iy >= mym1) iy = mym1;
if(iz < 0) iz = 0;  if(iz >= mzm1) iz = mzm1;
\end{lstlisting}
\end{remark}

\section{Force Evaluation}

For each particle, the tree correction computes the short-range gravitational force from nearby particles using a spline-softened kernel.  The opening angle criterion $\theta$ determines whether to use a cell's center of mass or open it:

\begin{equation}
  \frac{\ell}{r} < \theta \quad\Rightarrow\quad \text{use cell monopole}
\end{equation}

where $\ell$ is the cell size and $r$ is the particle--cell distance.  The opening angle is adaptively adjusted during the simulation:

\begin{lstlisting}[style=ccode,numbers=none]
if(stepcount < 10)       theta = 0.3;
else if(stepcount < 20)  theta = 0.35;
else if(stepcount < 60)  theta = 0.4;
else                     theta = 0.45;
\end{lstlisting}

\section{Type Conversion}

Before tree correction, particles are expanded in-place from 32-byte \code{pmparticletype} to 40-byte \code{treeparticletype} (\code{pm2treetype}), and shrunk back afterward (\code{tree2pmtype}).

\begin{algorithm}[pm2treetype --- In-Place Expansion]
\begin{enumerate}[leftmargin=2em]
  \item \code{Realloc} the particle array to $N_p \times 40$ bytes.
  \item Copy backwards (from last particle to first) using \code{memmove} to avoid overwriting:
\begin{lstlisting}[style=ccode,numbers=none]
for(i = nplong-1; i >= 0; i--) {
    memmove((char*)treeparticles + i*sizeof(treeparticletype),
            (char*)pmparticles + i*sizeof(pmparticletype),
            sizeof(pmparticletype));
}
\end{lstlisting}
  \item The \code{next} pointer (last 8 bytes) is left uninitialized and set during linked-list construction.
\end{enumerate}
\end{algorithm}


%======================================================
\chapter{Time Integration}
\label{ch:timeint}
%======================================================

\section{Leap-Frog Scheme}

The code uses a kick-drift-kick leap-frog integrator where the scale factor $a$ advances by a fixed step $\Delta a$:

\begin{align}
  v_{n+1/2} &= v_n + \frac{\Delta a}{2}\,\dot{v}_n \quad \text{(kick)} \\
  x_{n+1} &= x_n + \Delta a\,v_{n+1/2} \quad \text{(drift)} \\
  v_{n+1} &= v_{n+1/2} + \frac{\Delta a}{2}\,\dot{v}_{n+1} \quad \text{(kick)}
\end{align}

The velocity factors \code{fact1} and \code{fact2} encode the cosmological dependence:
\begin{equation}
  \ddot{a} = -\frac{4\pi}{3}\left(\frac{1}{a^2} + (1+3w)\frac{\Omega_\Lambda}{\Omega_m}\,\frac{f_\text{DE}(a)}{a^3}\right)
\end{equation}
computed by \code{appapsq()} in \file{FLRW.F}.

\section{Half-Step Synchronization}

The \code{halfstep} structure controls when positions and velocities are synchronized (both at the same time level) for output:
\begin{itemize}[leftmargin=2em]
  \item \code{halfstep.first = 1, second = 0}: First half-step (velocity kick only).
  \item \code{halfstep.first = 1, second = 1}: Second half-step (complete the full step).
\end{itemize}

This is triggered by output flags (\code{flagsyncpdata}) and the PreFoF group finder.

\chapter{Cosmology: CPL Dark Energy}
\label{ch:cpl}
%======================================================

\section{CPL Parameterization}

The equation of state for dark energy is parameterized as:
\begin{equation}
  w(a) = w_0 + w_a\,(1 - a/a_\text{max})
\end{equation}
where $a_\text{max}$ is the maximum scale factor.  The standard $\Lambda$CDM model corresponds to $w_0 = -1$, $w_a = 0$.

\section{Expansion History}

The Hubble parameter $H(a)$ in a flat universe:
\begin{equation}
  H^2(a) = H_0^2\left[\Omega_m\left(\frac{a_\text{max}}{a}\right)^3 + (1-\Omega_m-\Omega_\Lambda)\left(\frac{a_\text{max}}{a}\right)^2 + \Omega_\Lambda\,f_\text{DE}(a)\right]
\end{equation}
where
\begin{equation}
  f_\text{DE}(a) = \left(\frac{a}{a_\text{max}}\right)^{-3(1+w_0+w_a)}\exp\!\left[-3w_a\left(1 - \frac{a}{a_\text{max}}\right)\right]
\end{equation}
is the dark energy scaling function, implemented as \code{scaleDE()} in \file{FLRW.F}.

\section{Growth Factor}

The matter growth factor $D_+(a)$ satisfies:
\begin{equation}
  \ddot{D}_+ + 2H\dot{D}_+ = \frac{3}{2}\Omega_m(a)\,H^2\,C_w(a)\,D_+
\end{equation}
where $C_w(a) = 1 + (1+w)\,\Omega_\text{DE}(a)/\Omega_m(a)$ encodes the dark energy clustering.  This is solved numerically via the RK4 integrator in \code{growthcpl()} with the \code{odeint} driver from Numerical Recipes.

\section{Poisson Correction}

For dark energy models with clustering ($c_s = 0$), the Poisson equation is modified:
\begin{equation}
  \nabla^2\phi = 4\pi G\,\bar\rho_m\,\delta_m\,p_\text{corr}(a)
\end{equation}
where
\begin{equation}
  p_\text{corr}(a) = 1 + \frac{\Omega_\text{DE}(a)\,D_\text{DE}(a)}{\Omega_m(a)\,D_m(a)}
\end{equation}
This correction is computed at each time step by \code{PoissonCor()} in \file{FLRW.F} and applied in \code{pmmain.c}.

For smooth dark energy ($c_s = 1$), $p_\text{corr} = 1$ (no clustering correction).


%======================================================
\chapter{Domain Decomposition and Migration}
\label{ch:domain}
%======================================================

\section{Slab Decomposition}

The simulation box is divided into $N_\text{rank}$ slabs along the z-axis.  Each MPI rank owns a contiguous range of z-coordinates $[z_\text{min}, z_\text{max})$.  This aligns with the FFTW3-MPI data layout, which distributes the first (z) dimension across ranks.

\section{Particle Migration}

After positions are updated, particles that have moved outside their domain are exchanged via \code{migrate()} in \file{pmigrate.kisti.omp.c}:

\begin{algorithm}[Particle Migration]
\begin{enumerate}[leftmargin=2em]
  \item Scan all local particles; classify as staying, moving up, or moving down.
  \item Send ``up'' particles to rank+1, receive from rank$-$1 (periodic).
  \item Send ``down'' particles to rank$-$1, receive from rank+1 (periodic).
  \item Compact local array and append received particles.
  \item If multi-slab migration is needed, repeat for larger shifts.
\end{enumerate}
\end{algorithm}

\section{Load Balancing}

The domain rebalancer in \file{mydomaindecomposition.omp.c} monitors the particle count ratio:
\begin{equation}
  R = \frac{N_\text{max} - N_\text{min}}{N_\text{avg}} \times 100\%
\end{equation}
If $R > 1\%$ (the \code{MAXINHOMO} threshold), z-slab boundaries are adjusted so each rank holds approximately equal numbers of particles.

\chapter{Troubleshooting}
\label{ch:troubleshooting}
%======================================================

\section{Common Issues}

\subsection{Segfault in Tree Correction}

\textbf{Symptom}: Crash in \code{treecorrection()} or \code{ParticleCellLinkedList()}.

\textbf{Possible causes}:
\begin{enumerate}[leftmargin=2em]
  \item \textbf{Strict aliasing violation in \code{pm2treetype}}: At \code{-O2}, the compiler may reorder reads/writes of \code{pmparticles[i]} and \code{treeparticles[i]} because they are different types.  The fix is to use \code{memmove} through \code{char*} pointers.
  \item \textbf{Out-of-bounds cell index}: Particles near domain boundaries can produce negative or overly large cell indices.  Ensure clamping is applied.
  \item \textbf{Insufficient memory}: If \code{NMEG} is too small, the \code{Realloc} in \code{pm2treetype} fails.
\end{enumerate}

\subsection{Malloc Error (Extremely Large Allocation)}

\textbf{Symptom}: ``Error allocating memory \textit{N} bytes in Malloc()'' where \textit{N} is petabytes.

\textbf{Cause}: OpenMP clamping bug.  Check that the y and z index limits use \code{mym1}/\code{mzm1} (not \code{mxm1}) in \code{ParticleCellLinkedList()}.

\subsection{Power Spectrum File Not Found}

Ensure the \code{Ascii Powerfile} path in \file{params.dat} is relative to the run directory, not the source directory.

\section{Performance Tuning}

\begin{itemize}[leftmargin=2em]
  \item \textbf{NMEG}: Set large enough to hold all particles plus FFT workspace.  Rule of thumb: $\text{NMEG} \geq 40 \times N_p^{1/3}$ MB per rank for $N_p$ total particles.
  \item \textbf{MPI vs OpenMP balance}: Use fewer MPI ranks with more threads per rank to reduce communication overhead.  Typically 4--8 threads per rank.
  \item \textbf{Opening angle $\theta$}: Larger values are faster but less accurate.  The adaptive scheme (0.3--0.45) is a good default.
\end{itemize}

%======================================================
\chapter{FoF Halo Analysis}
\label{ch:fof}
%======================================================

At multiple time steps during the simulation run, the code dumps FoF halo data (particle positions and velocities).
The Friends-of-Friends algorithm identifies virialized halos using a linking length of $0.2$ times the mean particle separation.
The minimum halo mass is typically set to $m_h^c = 30 \times m_p$, where $m_p$ is the particle mass.

\section{Most Bound Particle (MBP)}

\begin{definition}[Most Bound Particle]
The MBP is the particle whose total energy is the most negative in a halo at the time the halo first forms.  Once assigned, the MBP persists as long as it remains among the member particles of some halo.  It serves as a proxy for the galaxy at the center of the halo.

The total energy of a particle is:
\begin{equation}
  E = \frac{1}{2}\,m\left(\frac{d\mathbf{r}}{dt}\right)^2 + \Phi(\mathbf{r})
\end{equation}
where the velocity includes both peculiar velocity and Hubble flow.  The gravitational potential is:
\begin{equation}
  \Phi(\mathbf{r}) = m\sum_i \frac{G\,m_i}{\sqrt{(\mathbf{r}_i - \mathbf{r})^2 + \epsilon^2}}
\end{equation}
where $\epsilon$ is the force resolution and $i$ runs over all member particles of the halo.
\end{definition}

At each time step, MBPs identified in the previous step are checked against the current FoF catalog:
\begin{itemize}[leftmargin=2em]
  \item If a halo contains no previously assigned MBP, a new MBP is selected by computing the total energy of all member particles.
  \item If one or more MBPs are already present, no new MBP is created.
\end{itemize}

\section{Attached Quantities}

The MBP is a particle with its own index, position, and peculiar velocity.  Since the MBP represents a galaxy within its halo, additional halo properties are attached:
\begin{itemize}[leftmargin=2em]
  \item Mean halo velocity $\mathbf{v}_h$
  \item Velocity dispersion $\sigma_v$
  \item Virial radius $r_\text{vir}$
  \item Halo mass (FoF mass)
\end{itemize}

\section{Center of Mass}

The center of mass (CoM) of a halo is the mean position of its member particles.  The group velocity is the average peculiar velocity.

\section{Halo Shape}

The shape of a halo is measured via the eigenvalues and eigenvectors of the shape tensor:
\begin{equation}
  T_{ij} = \sum_k r_{k,i}\,r_{k,j}
\end{equation}
where $\mathbf{r}_k$ is the displacement of particle $k$ from the halo CoM.  The eigenvalue ratios give the axis ratios of the halo.


%======================================================
\chapter{MBP Merger Tree}
\label{ch:mergertree}
%======================================================

The MBP merger tree tracks the hierarchical assembly of halos through cosmic time.  Each MBP carries a unique Global Identification number (GID) and a history of merger events.

\section{Basic Concepts}

\subsection{GID and Major GID}

\begin{definition}[GID]
The GID (Global Identification) is a unique number stored as \code{size\_t} in C format.  It is assigned when a halo first forms and its MBP is created.
\end{definition}

\begin{definition}[Major GID]
The Major GID is the GID of the most massive (central) MBP in a halo at each merger step.  It may change with time as the central MBP changes.
\end{definition}

\subsection{Merger Events}

The basic concept is to record each MBP's merger history in a single line.  Merger events are marked by a change in the Major GID:
\begin{itemize}[leftmargin=2em]
  \item When a halo first forms, it has one MBP whose Major GID equals its own GID.
  \item When halos merge, the MBP from the more massive progenitor becomes the \emph{central} (major) MBP; the other becomes a \emph{satellite} MBP.
  \item The satellite MBP's Major GID changes to the central MBP's GID.
\end{itemize}

\subsection{Central and Satellite MBP}

When two halos of different masses merge:
\begin{itemize}[leftmargin=2em]
  \item The MBP from the more massive halo is the \emph{central} MBP.
  \item The MBP from the less massive halo is the \emph{satellite} MBP.
  \item The satellite's mass, shape, and radius are frozen at their values just before merging.
  \item The satellite's $\sigma_v$ is set to the merger halo's $\sigma_v$.
  \item The satellite's position and velocity continue to be tracked via particle movement.
\end{itemize}

\subsection{MBP Mass}

The mass of an MBP equals its hosting halo mass when it is the sole or central MBP.  For satellite MBPs, the mass is frozen at the hosting halo mass measured just before the merger event.

\section{Data Structures}

\subsection{Original Merger Tree Format}

\begin{definition}[SavedHaloType]
\begin{lstlisting}[style=ccode,numbers=none]
typedef struct SavedHaloType {
    INDXTYPE mbp;                    /* MBP particle index */
    IDTYPE nowhid, majorglobalid;    /* halo ID, major GID */
    POSTYPE x, y, z;                 /* MBP position (h^-1 Mpc, comoving) */
    POSTYPE hx, hy, hz;              /* Halo CoM position */
    float mass, fofhmass;            /* MBP/halo mass, FoF halo mass (M_sun/h) */
    float vx, vy, vz;               /* MBP peculiar velocity (km/s) */
    float hvx, hvy, hvz;            /* Halo peculiar velocity (km/s) */
    short int statusflag, linkflag;  /* Internal flags */
} SavedHaloType;
\end{lstlisting}
\code{POSTYPE} is \code{double} when compiled with \flag{XYZDBL}, otherwise \code{float}.  \code{INDXTYPE} is typically \code{long}.
\end{definition}

\begin{important}
The field \code{majorglobalid} in \code{SavedHaloType} stores the Major GID, not a local halo index.
\end{important}

\subsection{GID-Sorted Merger Tree Format}

The sorted merger tree groups all tree lines belonging to the same $z=0$ halo together, enabling efficient analysis of merger histories.

\begin{definition}[GID-Sorted Data Structures]
\begin{lstlisting}[style=ccode,numbers=none]
typedef struct MajorGid {
    IDTYPE gid, newgid, mgid;
    IDTYPE majorgid[NSTEPS];
} MajorGid;

typedef struct GID {
    IDTYPE gid, newgid;
} GID;

typedef struct SigV {
    float sigv[NSTEPS];
} SigV;

typedef struct GidMergeHistory {
    IDTYPE gid;
    SavedHaloType step[NSTEPS];
    SigV sigv;
} GidMergeHistory;
\end{lstlisting}
Here \code{IDTYPE} is \code{long} and \code{NSTEPS} is the number of merger tree time steps.
\end{definition}

The output file contains:
\begin{enumerate}[leftmargin=2em]
  \item A header with simulation parameters (box size, $h$, $n_s$, $\Omega_m$, $\Omega_b$, $\Omega_\Lambda$, $\sigma_8^{-1}$, $\Delta a$, number of merger steps, step numbers, and redshifts).
  \item An array of \code{GidMergeHistory} structures containing the full merger history.
\end{enumerate}

\section{Building the Merger Tree}

\begin{algorithm}[Merger Tree Construction]
\begin{enumerate}[leftmargin=2em]
  \item \textbf{Extract halo properties}: Run the FoF halo finder to produce halo catalogs.
\begin{lstlisting}[style=bash,numbers=none]

\end{lstlisting}
  This produces files including \file{FoFHaloQuantities.\%.5d.bin}.

  \item \textbf{Link MBPs between steps}: Connect MBPs across adjacent time steps.
\begin{lstlisting}[style=bash,numbers=none]
mpirun -np <N> ./sim2link.mpi.mod2.exe <step_before> <step_after>
\end{lstlisting}

  \item \textbf{Build merger chains}: Assemble the linked MBPs into complete merger trees.
\begin{lstlisting}[style=bash,numbers=none]
mpirun -np <N> ./mkmerger.exe
\end{lstlisting}

  \item \textbf{Combine output}: Merge all partial files into a single merger tree file.
\begin{lstlisting}[style=bash,numbers=none]
makeone
\end{lstlisting}
\end{enumerate}
\end{algorithm}

\section{Sorting by Major GID}

The merger tree lines are sorted by their Major GID at the final redshift (typically $z=0$), so that all tree lines belonging to the same $z=0$ halo are grouped together.

\begin{algorithm}[GID Sorting]
\begin{enumerate}[leftmargin=2em]
  \item \textbf{Assign new GIDs}: Re-index the merger tree by major GID.
\begin{lstlisting}[style=bash,numbers=none]
mpirun -np <N> ./SortMajorGid.exe <redshift>
\end{lstlisting}
  Source files: \file{SortMajorGid.c}, \file{pqsort.SMG.c}, \file{SMG.h}.

  \item \textbf{Sort and dump}: Produce the final sorted output file.
\begin{lstlisting}[style=bash,numbers=none]
mpirun -np <N> ./Sort_SortMajorGid.exe <input_header> <num_files>
\end{lstlisting}
  Source files: \file{Sort\_SortMajorGid.c}, \file{pqsort.SMG.c}, \file{SMG.h}.
\end{enumerate}
\end{algorithm}

\begin{remark}[Internal Parameters]
The internal structure of the sorted merger tree depends on the final redshift.  When building, check the values of \code{iredshift}, \code{NSTEPS}, and \code{MSTEPS} in the source code to match your simulation.
\end{remark}

\section{Known Issues}

\subsection{Disappearance of MBP}

In rare cases, an MBP may be ejected from its host halo (e.g., into the intergalactic medium).  When this happens, the MBP line terminates and its variables are set to null for subsequent steps.  A new MBP is then selected in the halo.

\subsection{Fluctuating Small Halos}

When a halo's mass is near the critical threshold, numerical fluctuations can cause it to drop below the threshold at the next step.  This may generate multiple MBPs for a single physical halo; only one will evolve further while the others terminate.

\subsection{Null Values}

A null, zero, or negative value in the \code{mbp} field indicates that the MBP was not found at that epoch.

\subsection{Mass Conservation}

The total mass before and after a merger event may differ due to mass loss during the merger process.  Analysis should account for this discrepancy.

\chapter{PSB Subhalo Finder}
\label{ch:psb}
%======================================================

The PSB (Phase-Space Based) subhalo finder is a standalone MPI post-processing tool located in \file{SRC(FoF\_PSB\_Free)/}.  It identifies gravitationally bound substructures within FoF halos by combining density peak detection, contour-based region separation, and iterative energy unbinding.  The executable \code{hfind.exe} reads the FoF output produced in Chapter~\ref{ch:fof} and writes a subhalo catalog for each time step.

\begin{remark}[Standalone Binary]
\code{hfind.exe} is compiled and run independently from the main simulation.  It requires only the FoF halo catalog and member particle files as input.
\end{remark}

\section{Algorithm Overview}

\begin{algorithm}[PSB Subhalo Identification Pipeline]
The subhalo finder operates in six stages:
\begin{enumerate}[leftmargin=2em]
  \item \textbf{Density estimation}: Compute the SPH density at each particle position using an adaptive grid and the W4 cubic spline kernel.
  \item \textbf{Peak detection}: Identify local density maxima using a 26-neighbor criterion.
  \item \textbf{Density contour separation}: For each pair of neighboring peaks, perform a binary search in density to find the saddle-point contour that separates them.  Use depth-first-search (DFS) flood-fill to assign particles to each peak's region.
  \item \textbf{Energy unbinding}: Remove kinetically unbound particles via iterative tidal-radius and total-energy checks.
  \item \textbf{FoF substructure linking}: Apply tree-accelerated Friends-of-Friends linking to associate remaining shell particles with the nearest bound core.
  \item \textbf{Output}: Write the subhalo catalog and background (unbound) particles.
\end{enumerate}
\end{algorithm}

\section{Density Estimation}

The density field is computed on a sub-grid using the SPH W4 cubic spline kernel.

\begin{definition}[SPH W4 Cubic Spline Kernel]
The smoothing kernel used for density estimation is:
\begin{equation}
  W(r,h) = \frac{1}{\pi h^3} \begin{cases}
    1 - \tfrac{3}{2}q^2 + \tfrac{3}{4}q^3, & 0 \le q < 1 \\[4pt]
    \tfrac{1}{4}(2-q)^3, & 1 \le q < 2 \\[4pt]
    0, & q \ge 2
  \end{cases}
\end{equation}
where $q = r/h$ and $h$ is the smoothing length.  The implementation is in \file{find\_density.f}.
\end{definition}

The density grid has a maximum size of \code{MAX\_BOX\_MEMORY} $= 421{,}875{,}000$ cells ($750^3$).  The grid dimensions are adapted to the bounding box of each FoF halo's member particles.  MPI-parallel density estimation exchanges boundary layers (\code{nbuff}${}=5$ cells) between adjacent ranks via \code{MPI\_Sendrecv}.

\begin{algorithm}[Density Field Construction]
\begin{enumerate}[leftmargin=2em]
  \item Compute bounding box of all particles in the halo.
  \item Determine sub-grid cell size $h$ and grid dimensions $(n_x, n_y, n_z)$.
  \item Assign each particle to the nearest grid cell.
  \item For each particle, deposit its contribution to neighboring cells using $W(r,h)$.
  \item Exchange boundary layers between MPI ranks.
  \item Interpolate the grid density back to particle positions.
\end{enumerate}
\end{algorithm}

\section{Peak Detection}

\begin{algorithm}[Density Peak Detection]
\begin{enumerate}[leftmargin=2em]
  \item For each particle $i$ with density $\rho_i > \code{peakthreshold}$, examine all 26 neighbors in the density grid (the $3^3 - 1$ surrounding cells).
  \item If $\rho_i$ is strictly greater than all 26 neighbors, mark particle $i$ as a density peak and set the \code{WP\_PEAK} flag.
  \item Peaks with fewer than \code{MIN\_HALO\_SEED\_NUM}${}=10$ particles in their immediate neighborhood are discarded.
\end{enumerate}
The maximum number of peaks per halo is \code{MAX\_PEAK\_NUM}${}=100{,}000$.
\end{algorithm}

\begin{infobox}[Peak Threshold]
The parameter \code{peakthreshold}${}=1.5$ (set in \file{sub\_find\_dimension.inc}) controls the minimum overdensity required for a local maximum to be considered a density peak.  Lower values produce more candidate peaks but increase computation time.
\end{infobox}

\section{Density Contour Separation}

Once density peaks are identified, the algorithm must separate overlapping regions.  For each pair of adjacent peaks, a binary search in density threshold locates the saddle-point contour.

\begin{algorithm}[Contour Separation via Binary Search]
\begin{enumerate}[leftmargin=2em]
  \item For each peak, find the nearest neighboring peak.
  \item Set the initial density search range: $[\rho_\text{low}, \rho_\text{high}] = [\code{downthreshold}, \min(\rho_{\text{peak},1}, \rho_{\text{peak},2})]$.
  \item Bisect: set $\rho_\text{mid} = (\rho_\text{low} + \rho_\text{high})/2$.
  \item Perform a DFS flood-fill from each peak, visiting only cells with $\rho > \rho_\text{mid}$.
  \item If the two flood-fills meet (regions are connected), raise $\rho_\text{low} = \rho_\text{mid}$; otherwise lower $\rho_\text{high} = \rho_\text{mid}$.
  \item Repeat until $|\rho_\text{high} - \rho_\text{low}| < \code{exactthreshold} = 0.2$.
\end{enumerate}
The converged $\rho_\text{mid}$ defines the core density threshold for each peak.
\end{algorithm}

\begin{definition}[DFS Flood-Fill]
Starting from a density peak, the depth-first search visits neighboring cells on the packed 3D grid.  A cell is visited if its density exceeds the current threshold $\rho_\text{mid}$ and it has not been previously visited (checked via the \code{WP\_VISITED} flag).  The maximum recursion depth is \code{ndepth}${}=10{,}000{,}000$.
\end{definition}

\section{Energy Unbinding}

Particles in each core and surrounding shell are tested for gravitational binding.  Unbound particles are iteratively removed.

\begin{algorithm}[Iterative Energy Unbinding]
\begin{enumerate}[leftmargin=2em]
  \item Compute the tidal radius $R_\text{tidal}$ for each core from its mass and the background density.
  \item For each particle within $R_\text{tidal}$, compute:
    \begin{itemize}[leftmargin=2em]
      \item Kinetic energy: $K = \frac{1}{2} m |\mathbf{v} - \mathbf{v}_\text{cm}|^2$
      \item Gravitational potential: $\Phi = -G \sum_j m_j / \sqrt{r_{ij}^2 + \epsilon^2}$
    \end{itemize}
  \item Remove particles with $E = K + \Phi > 0$ (unbound).
  \item Recompute the center of mass and tidal radius.
  \item Repeat for \code{BOUND\_ITER}${}=4$ iterations.
\end{enumerate}
Cores are processed in order of increasing mass (smallest first).
\end{algorithm}

\begin{important}
The ordering of cores during unbinding matters.  Processing smaller cores first prevents them from being absorbed into larger neighbors before their own binding state is evaluated.
\end{important}

The potential calculation uses the spline-softened force kernel initialized from \file{force\_spline.mod3.c}.  The number of integration sub-steps is \code{NSTEP}${}=5$, and the potential is averaged over \code{REPEAT\_POTENT\_NUM}${}=4$ evaluations.

\section{FoF Substructure Linking}

After unbinding, remaining shell particles are associated with cores via tree-accelerated Friends-of-Friends linking.

\begin{algorithm}[Tree-Based FoF Linking]
\begin{enumerate}[leftmargin=2em]
  \item Build a spatial tree over all core and shell particles.
  \item For each particle, walk the tree to find neighbors within the linking length $b = \code{FOF\_LINK} \times \bar{d}$, where $\bar{d}$ is the mean particle separation and $\code{FOF\_LINK} = 0.2$.
  \item If two particles are linked, merge their group labels.
  \item Iterate until no new links are formed.
  \item Assign each FoF group to the core it contains (or the nearest core if none).
\end{enumerate}
The tree walk is implemented in \file{fof.mod8.F} using the tree structures from \file{Treewalk.hfind.mod2.c}.
\end{algorithm}

\section{MPI Parallelization}

The code uses a master-worker pattern.  Rank~0 reads the FoF catalog and distributes individual halos to worker ranks for subhalo analysis.

\begin{algorithm}[Master-Worker Communication Protocol]
\begin{enumerate}[leftmargin=2em]
  \item \textbf{Broadcast}: Rank~0 broadcasts cosmological parameters (box size, $h$, $\Omega_m$, $\Omega_\Lambda$, scale factor) to all ranks.
  \item \textbf{Distribution}: For each halo in the catalog:
    \begin{enumerate}
      \item Rank~0 probes for a \code{READY} message from any worker.
      \item Rank~0 sends the particle count (\code{NP\_TAG}), positions (\code{R\_TAG}), and indices (\code{INDX\_TAG}) to the ready worker.
    \end{enumerate}
  \item \textbf{Processing}: Each worker calls \code{subhalo\_den()} on the received halo data.
  \item \textbf{Collection}: The worker sends back the particle-to-halo mapping (\code{PT2H\_TAG}) and the number of subhalos found (\code{MPEAK\_TAG}).
  \item \textbf{Output}: Rank~0 writes the results for each halo to disk (\code{WRITING} tag signals write phase).
\end{enumerate}
\end{algorithm}

\section{Input and Output Formats}

\subsection{Input Files}

The subhalo finder reads two binary files produced by the FoF halo finder (Chapter~\ref{ch:fof}):
\begin{itemize}[leftmargin=2em]
  \item \file{FoF\_halo\_cat.NNNNN} --- FoF halo catalog header containing simulation parameters (box size, $h$, $\Omega_m$, $\Omega_b$, $\Omega_\Lambda$, $w_0$, $w_a$, grid size, scale factor) and the number of halos.
  \item \file{FoF\_member\_particle.NNNNN} --- Binary dump of member particles for each FoF halo (positions, velocities, and indices as \code{basicparticletype}).
\end{itemize}

\begin{remark}[Unit Conventions]
Positions in the input files are in grid units (integer grid index).  The code converts to physical units ($h^{-1}\,\text{Mpc}$, comoving) using the box size and grid dimensions read from the catalog header.  Velocities are stored as simulation-internal values and converted to km\,s$^{-1}$ via the scale-factor-dependent conversion factors \code{r1kineticfact} and \code{r2kineticfact}.
\end{remark}

\subsection{Output Files}

\begin{itemize}[leftmargin=2em]
  \item \file{HFIND.DAT.NNNNN} --- Binary subhalo catalog.  For each FoF halo, the file contains:
    \begin{enumerate}[leftmargin=2em]
      \item \code{int mpeak}: number of subhalos found.
      \item For each subhalo $i = 0, \ldots, \code{mpeak}-1$:
        \begin{itemize}[leftmargin=2em]
          \item \code{int wnp}: number of member particles.
          \item \code{double x,y,z} $\times$ \code{wnp}: positions (physical units).
          \item \code{double vx,vy,vz} $\times$ \code{wnp}: velocities (km\,s$^{-1}$).
          \item \code{indxtype indx} $\times$ \code{wnp}: particle indices.
        \end{itemize}
    \end{enumerate}
  \item \file{background\_ptl.NNNNN} --- Particles not bound to any subhalo (same binary format as above, without subhalo grouping).
\end{itemize}
